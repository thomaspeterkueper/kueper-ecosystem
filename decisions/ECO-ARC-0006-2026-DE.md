# ECO-ARC-0006-2026-DE — Kanonisches External-Task-Format

Status: accepted
Datum: 2026-07-12
Scope: systemweit
Betroffene Repositories: `kueper-ecosystem`, alle registrierten KUEPER-Repositories

## Kontext

Der Cross-Repository-Workflow (`docs/cross-repository-workflow.md`) definiert seit
Beginn ein empfohlenes Dateinamens- und Inhaltsschema fuer External Tasks. In der
Praxis (Audit 2026-07-12) folgte jedoch **keine** der 16 offenen External Tasks
diesem Schema. Stattdessen existierten mindestens fuenf inkompatible Formate:

- `NOX-####` (NOXIA)
- `SSF-####` mit `Status`/`Created`/`## Origin`/`## Reason`/`## Requested Change`
- `KB-REQUEST`-Block (Knowledge Graph, key-per-Zeile)
- `REQ-KG-*` mit HTML-Kommentar-Kopf
- diverse `# External Task`- und `# KG request`-Freiformen (OTA)

Folgen: keine maschinelle Auswertung, keine automatisierte Zaehlung/Weiterleitung,
kein zuverlaessiger Lebenszyklus, keine Collector-Integration.

## Entscheidung

Es gilt genau **ein** kanonisches External-Task-Format. Es besteht aus einem
YAML-Frontmatter (maschinenlesbar) und einem Markdown-Body mit festen
Pflicht-Sektionen.

### Dateiname

```text
EXT-{SOURCE}-{TARGET}-{YYYYMMDD}-{NNN}.md
```

### System-Codes

| Code | Repository |
|---|---|
| `ECO` | `kueper-ecosystem` |
| `KG` | `kueper-knowledge-graph` |
| `SSF` | `solarsciencefoundation` |
| `NOXIA` | `noxiagame` |
| `NXU` | `noxia-universe` |
| `MISH` | `mishkenaz.org` |
| `OMNI` | `omnizedenz.org` |
| `AVI` | `avi-modell` |
| `CONTRA` | `contracomology` |
| `ARCH` | `kueper-archive-schema` |
| `KUE` | `kueper.com` |
| `OTA` | `overtime-archive.org` |
| `TKD` | `thomas-kueper.de` |

`SOURCE` ist das anfordernde, `TARGET` das zustaendige (Inbox-)Repository.
`SOURCE` und `TARGET` duerfen nicht identisch sein.

### Frontmatter

```yaml
---
id: EXT-SSF-KG-20260712-001
title: Kurztitel
status: open            # open | done | rejected | parked
source: SSF
target: KG
created: 2026-07-12
requested_by: T.P.K.
priority: high          # low | medium | high | critical (optional)
affects: [KG, SSF]      # optional
supersedes: []          # optional, Liste alter IDs
---
```

### Pflicht-Sektionen im Body

```md
## Anlass
## Gewünschte Änderung
## Begründung
## Betroffene Repositories
## Erwartetes Ergebnis
## Hinweise            (optional)
```

Validiert durch `schemas/external-task.schema.json` und
`tools/lint-external-tasks/lint.py`.

## Lebenszyklus

`open` -> Bearbeitung im Ziel-Repository -> Datei wird nach
`external-tasks/done/`, `rejected/` oder `parked/` verschoben, und `status`
im Frontmatter wird entsprechend gesetzt. Der Dateiname bleibt stabil; er ist
die dauerhafte Referenz-ID.

## Migration der Bestandsformate

Bestehende Tasks werden nicht rueckwirkend erzwungen umgeschrieben. Es gilt:

1. Neue Tasks folgen ab sofort ausschliesslich diesem Format.
2. Wird ein Bestands-Task bearbeitet, wird er beim Statuswechsel in das
   kanonische Format ueberfuehrt (Rename + Frontmatter), bevor er nach
   `done/`/`rejected/` wandert.
3. Feld-Mapping der haeufigsten Altformate:

| Alt | Kanonisch |
|---|---|
| `## Origin` / `## Purpose` / `Origin:` | `## Anlass` |
| `## Requested Change` / `Requested Content:` | `## Gewünschte Änderung` |
| `## Reason` | `## Begründung` |
| `## Target` / `Recipient:` / `Target System` | Frontmatter `target` |
| `Requester:` / `Origin System` | Frontmatter `source` |
| `## Blocking` | Frontmatter `priority` + `## Hinweise` |
| `Priority:` | Frontmatter `priority` |

## Auswirkungen

- Der Collector (ECO-ARC-0005) kann offene Tasks maschinell zaehlen, nach
  Ziel/Prioritaet auswerten und Abweichungen melden.
- Jede Task ist eindeutig quell- und zielbezogen identifizierbar.
- Ein CI-Linter kann Format-Verstoesse vor dem Merge blockieren.

## Nicht entschieden

- Ob der Linter als GitHub-Action in jedem Ziel-Repo verpflichtend wird
  (separate External Tasks pro Repo noetig, da Fremd-CI-Konfiguration).
- Automatisches Umschreiben aller 16 Bestands-Tasks in einem Rutsch.

## Folgeaufgaben

1. Template unter `docs/templates/external-task-template.md` bereitstellen. (erledigt)
2. Linter unter `tools/lint-external-tasks/` bereitstellen. (erledigt)
3. Ziel-Repositories per External Task bitten, den Linter als CI-Check zu uebernehmen.
