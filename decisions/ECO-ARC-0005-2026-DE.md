# ECO-ARC-0005-2026-DE — Status- und Monitoring-Control-Plane

Status: accepted  
Datum: 2026-07-10  
Scope: systemweit  
Betroffene Repositories: `kueper-ecosystem`, perspektivisch alle registrierten KUEPER-Repositories

## Kontext

Mit wachsender Zahl eigenständiger Projekte reicht eine rein manuelle Übersicht über Repository-Stände, Deployments, Konfigurationen und Integrationen nicht mehr aus.

Jedes Projekt bleibt Source of Truth für seinen eigenen fachlichen und technischen Zuständigkeitsbereich. Das `kueper-ecosystem` soll diese Zuständigkeiten nicht übernehmen, sondern den gemeldeten und beobachteten Zustand der Projekte zusammenführen, bewerten und überwachen.

## Entscheidung

Das `kueper-ecosystem` wird als zentrale **Control Plane** für Status und Monitoring des KUEPER-Ökosystems aufgebaut.

Es hat vier Kernaufgaben:

1. **Stand liefern** — aktueller Zustand aller registrierten Projekte.
2. **Abweichungen erkennen** — fehlende, veraltete oder widersprüchliche Zustände markieren.
3. **Verbindungen überwachen** — Erreichbarkeit und Kompatibilität zwischen Projekten prüfen.
4. **Handlungsbedarf ableiten** — Warnungen erzeugen und bei repository-fremden Änderungen External Tasks im zuständigen Ziel-Repository anfordern.

Das Ecosystem speichert keine fachlichen Projektdaten als eigene Wahrheit. Es speichert ausschließlich:

- Registrierungs- und Beobachtungskonfiguration,
- zuletzt ermittelte Status-Snapshots,
- Prüfergebnisse,
- Warnungen,
- historische Statuswechsel,
- Referenzen auf zuständige Repositories und External Tasks.

## Statusmodell

Jedes registrierte Projekt liefert beziehungsweise erhält einen normalisierten Status in folgenden Bereichen.

### 1. Project

- Repository vorhanden und erreichbar
- Default Branch
- aktueller Commit
- deklarierte Projektversion
- letzter Commit-Zeitpunkt
- offene Pull Requests
- offene External Tasks
- vorhandene Status- oder Manifestdatei

### 2. Build und Deployment

- letzter Build
- Build-Status
- Deployment-Umgebung
- Deployment-URL
- Domain-Status
- Zeitpunkt des letzten erfolgreichen Deployments

### 3. Runtime Health

- Anwendung erreichbar
- Health-Endpunkt erreichbar
- Antwortzeit
- API-Version
- Schema-Version
- Cron- oder Hintergrundjob-Status, sofern vorhanden

### 4. Integration

- erwartete Verbindung
- tatsächliche Erreichbarkeit
- verwendete API-Version
- erwartete API-Version
- Schema-Kompatibilität
- letzter erfolgreicher Integrationscheck

### 5. Configuration

- erforderliche Umgebungsvariablen vorhanden
- optionale Umgebungsvariablen vorhanden
- Secret-Referenzen vorhanden
- keine Secret-Werte im Ecosystem gespeichert
- Konfigurationsversion kompatibel

### 6. Governance

- README vorhanden
- Versionsangabe vorhanden
- erforderliche Dokumentation vorhanden
- External-Task-Struktur vorhanden
- Source-of-Truth-Regeln eingehalten
- weitere systemweite Policies erfüllt

## Zustandswerte

Jeder Einzelcheck verwendet einen der folgenden Werte:

- `ok` — Prüfung erfolgreich
- `warning` — funktionsfähig, aber veraltet, unvollständig oder risikobehaftet
- `error` — erforderlich und nicht funktionsfähig
- `unknown` — noch nicht geprüft oder nicht eindeutig ermittelbar
- `not_applicable` — für dieses Projekt nicht relevant

Der Gesamtstatus eines Projekts wird aus den Einzelchecks abgeleitet:

- `healthy` — keine Warnungen oder Fehler
- `degraded` — mindestens eine Warnung, kein Fehler
- `critical` — mindestens ein Fehler
- `unknown` — keine hinreichenden Prüfdaten

Ein grüner Gesamtstatus darf niemals aus fehlenden Daten abgeleitet werden. Fehlende Prüfdaten bleiben `unknown`.

## Registrierungsprinzip

Projekte werden deklarativ im Ecosystem registriert. Die Registry enthält nur Informationen, die zur Beobachtung notwendig sind, beispielsweise:

```yaml
id: noxia
name: NOXIA
repository: thomaspeterkueper/noxiagame
provider: github
production_url: https://example.invalid
health_url: https://example.invalid/api/health
version_source:
  type: package-json
  path: package.json
integrations:
  - target: ssf
    required_api: "^2.0"
  - target: knowledge-graph
    required_schema: "^1.8"
required_environment:
  - SSF_BASE_URL
  - KNOWLEDGE_SOURCE
```

Die Registry beschreibt Erwartungen. Der Collector ermittelt den Ist-Zustand. Die Auswertung vergleicht Soll und Ist.

## Komponenten

### Registry

Deklarative Liste aller überwachten Projekte, Integrationen und Erwartungen.

### Collector

Liest Zustände aus GitHub, Vercel, Health-Endpunkten und später weiteren Providern ein.

### Normalizer

Überführt provider-spezifische Daten in das gemeinsame Statusmodell.

### Evaluator

Vergleicht Soll- und Ist-Zustand und erzeugt `ok`, `warning`, `error` oder `unknown`.

### Snapshot Store

Speichert den letzten normalisierten Zustand und Statuswechsel. Er ist kein Ersatz für die Datenquelle.

### Alert Engine

Erzeugt Warnungen bei neuen oder verschärften Abweichungen. Wiederholt unveränderte Warnungen nicht unnötig.

### Dashboard/API

Liefert den aktuellen Gesamtstand und Detailansichten. Es enthält keine eigene Bewertungslogik, sondern zeigt Ergebnisse des Evaluators.

## Erster Umsetzungsumfang

Die erste funktionsfähige Version überwacht ausschließlich GitHub-basierte Zustände:

1. registrierte Repositories
2. Default Branch und letzter Commit
3. deklarierte Projektversion
4. offene Pull Requests
5. offene Dateien unter `external-tasks/open/`
6. letzter CI-Status, soweit über GitHub Actions verfügbar
7. Vorhandensein definierter Governance-Dateien

Vercel, Domains, Environment-Prüfung, Runtime-Health und Kompatibilitätschecks folgen als getrennte Erweiterungen.

## Datenschutz und Secrets

Das Ecosystem speichert niemals Secret-Werte.

Es darf ausschließlich prüfen und anzeigen:

- Secret oder Variable vorhanden: ja/nein
- Scope oder Umgebung
- letzter Änderungszeitpunkt, sofern verfügbar
- Referenzname

Logs und Snapshots dürfen keine Tokens, Schlüssel oder vollständigen sensiblen Header enthalten.

## External Tasks

Erkennt das Ecosystem eine notwendige Änderung in einem anderen Repository, führt es diese nicht fachlich selbst aus.

Stattdessen wird entsprechend `docs/cross-repository-workflow.md` im zuständigen Ziel-Repository eine Anforderung unter `external-tasks/open/` angelegt.

Das Ecosystem darf eigene Collector-, Registry-, Evaluator- und Dashboard-Änderungen direkt im Repository `kueper-ecosystem` umsetzen.

## Nicht entschieden

- konkrete Persistenztechnologie für Snapshots
- konkrete Hosting-Plattform der Control Plane
- Authentifizierungsmodell des Dashboards
- Benachrichtigungskanäle
- automatische Erstellung von External Tasks ohne manuelle Freigabe
- endgültige Frequenz der Checks

## Folgeaufgaben

1. Projekt-Registry und Schema anlegen.
2. NOXIA, SSF, Knowledge Graph, OTA und `kueper.com` registrieren.
3. GitHub-Collector als ersten Provider implementieren.
4. normalisiertes Snapshot-Schema definieren.
5. Evaluator und Gesamtstatus ableiten.
6. erste Statusübersicht als API und einfache Seite bereitstellen.
7. Vercel-Integration separat spezifizieren.
