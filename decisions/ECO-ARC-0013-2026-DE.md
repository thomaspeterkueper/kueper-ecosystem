# ECO-ARC-0013-2026-DE — KUEPER Archive Core: gemeinsames Datenbankschema für Dokumentarchive

Status: accepted
Datum: 2026-07-14
Scope: `kueper.com`, `overtime-archive.org`, neues Repository `kueper-archive-schema`
Betroffene Repositories: `kueper-archive-schema` (neu), `kueper.com`, `overtime-archive.org`, `kueper-knowledge-graph`

## Kontext

Für `kueper.com` (KUE, Namespace `DOC:KUE:*`) und `overtime-archive.org` (OTA,
Namespace `DOC:OTA:*`) wurde ein detaillierter Vorschlag für ein Supabase-
gestütztes Dokumentarchiv-Schema vorgelegt (Owner, 2026-07-14). Der Vorschlag
selbst identifiziert die zentrale offene Frage bereits korrekt und schlägt vor,
sie über einen External Task an `ECO` zu klären, bevor produktiv etwas angelegt
wird:

> „Da dieses Schema sowohl KUE als auch OTA betrifft, gehört seine kanonische
> Zuständigkeit nicht automatisch allein in eines der beiden Repositories."

Diese Entscheidung beantwortet das.

## Entscheidung

### 1. Architektur wird bestätigt: identisches Schema, getrennte Datenbanken

Die vorgeschlagene Grundentscheidung wird übernommen:

- **Ein gemeinsames Datenmodell** („KUEPER Archive Core"), **zwei physisch
  getrennte Supabase-Projekte** (eines für KUE, eines für OTA).
- Keine gemeinsame Datenbank mit `tenant_id`/`archive_id`-Mandantenlogik.
- Gemeinsamkeit entsteht durch identische Migrationen, Typdefinitionen,
  Importer, Validierungsregeln und Admin-Oberfläche — nicht durch geteilte
  Datenhaltung.
- Vier Schichten: kanonische Dokumentidentität, inhaltliche Nachweise
  (Claims/Sources), Redaktion/Workflow, Suche/Synchronisation.
- Diese Architektur ist mit ECO-ARC-0007 (KG = Identität, KUE/OTA = Inhalt)
  konsistent und konkretisiert sie auf Datenbankebene: Die Archivdatenbanken
  halten redaktionelle Dokumentbeziehungen; das KG bleibt zuständig für
  globale Entitätsidentitäten und projektübergreifende Mappings.

### 2. Zuständigkeit: eigenes, zentral verantwortetes Repository

Das Schema erhält ein **eigenes, registriertes Infrastruktur-Repository**:
`kueper-archive-schema`. Es gehört weder `kueper.com` noch
`overtime-archive.org` allein — beide sind gleichberechtigte Konsumenten.

```text
kueper-ecosystem       entscheidet und dokumentiert den Standard (diese Datei)
kueper-archive-schema  besitzt Migrationen, Typdefinitionen, Importer, Validatoren
kueper.com / OTA       wenden die Migrationen in ihrer jeweils eigenen,
                        physisch getrennten Supabase-Instanz an
```

Rolle: `infrastructure` (neu ergänzt in `schemas/project-registry.schema.json`).
Ökosystem-Code: `ARCH`.

### 3. Schema V1 wird als Ausgangspunkt übernommen

Der vorgelegte SQL-Entwurf (Phase 1: Archivregister, Phase 2: Redaktion,
Phase 3: Wissensnachweise) wird als **Schema V1, Status Entwurf** in
`kueper-archive-schema` abgelegt — vollständig transkribiert, nicht neu
entworfen. Er ist damit dokumentiert und versioniert, aber **noch nicht auf
eine reale Supabase-Instanz angewendet**. Das Anwenden bleibt Aufgabe von
`kueper.com` bzw. `overtime-archive.org` (siehe Umsetzungsplan unten).

### 4. Was zwischen KUE und OTA identisch bleibt vs. konfiguriert wird

Wie im Vorschlag spezifiziert: Tabellen, Migrationen, RLS, Import-/
Validierungslogik, Sync-Protokoll, Admin-Oberfläche, Quellen-/Claim-Modell,
Request-Workflow und Suchindexstruktur sind identisch. Konfiguriert wird pro
Instanz nur `archive_config` (Archivcode, System-ID, ID-Präfix, Repository,
Dokumentklassen, Evidenzmarker-Profil, kanonische URL, inhaltlicher Scope).

### 5. Sicherheitsleitplanke: Sync-Richtung

`sync_runs.direction` beginnt ausschließlich als `github_to_database`. Eine
kontrollierte Gegenrichtung (`database_to_branch`) kann später folgen, aber
**niemals** `database_to_main` — die Datenbank wird nicht zur Schreibquelle für
den Hauptbranch. Diese Leitplanke ist verbindlich, nicht nur Empfehlung.

## Auswirkungen

- Neues Repository `kueper-archive-schema`, registriert mit Rolle
  `infrastructure`, Code `ARCH`, Integrationen zu `kueper-com`, `ota` und
  `knowledge-graph`.
- `schemas/project-registry.schema.json` um die Rolle `infrastructure`
  erweitert.
- Schema V1 (Migrationen, Seeds für KUE/OTA) liegt dokumentiert im neuen
  Repository, Status Entwurf/nicht angewendet.

## Nicht entschieden

- Konkreter Zeitpunkt der produktiven Anwendung auf KUE bzw. OTA.
- Feingranulare Rollen/Teams über das im Vorschlag skizzierte
  `owner/editor/reviewer/viewer`-Minimalmodell hinaus.
- Ob und wie `database_to_branch` später umgesetzt wird.
- Sprachabhängige Suchkonfiguration (`german`/`english`/`simple`) über die im
  Vorschlag benannte Einschränkung hinaus.

## Umsetzungsplan (aus dem Vorschlag übernommen)

1. Schema V1 verbindlich dokumentieren. (erledigt — dieses Repository)
2. Zuständigkeit klären. (erledigt — diese Entscheidung)
3. Gemeinsame Migrationen anlegen. (erledigt — Schema V1 im neuen Repository)
4. Zuerst KUE importieren und testen. (offen, Aufgabe von `kueper.com`)
5. Dieselben Migrationen unverändert auf OTA anwenden. (offen, Aufgabe von
   `overtime-archive.org`, erst nach Schritt 4)

## Folgeaufgaben

1. Registry-Eintrag `kueper-archive-schema`. (erledigt)
2. Repository mit Schema V1 (Migrationen, Seeds, Architektur-Dokumentation)
   anlegen. (erledigt)
3. External Task an `kueper.com`: Schema V1 prüfen und als erstes von beiden
   Archiven importieren/testen (Umsetzungsplan Schritt 4). (erledigt)
4. External Task an `overtime-archive.org`: nach Abschluss von Schritt 4
   dieselben Migrationen unverändert übernehmen (Umsetzungsplan Schritt 5).
   (erledigt, abhängig von Aufgabe 3)
